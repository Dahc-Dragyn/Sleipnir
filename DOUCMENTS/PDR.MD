Product Requirements Document (PRD)Project Sleipnir: The High-Speed Agent Orchestration Workbench1. Purpose & ScopeProject Sleipnir is a lightning-fast, terminal-based (ratatui-driven) orchestration dashboard and control runtime written in Rust. It acts as a centralized "Human-on-the-Loop" tactical command center for autonomous AI agents (including those built with Neuroweave, Antigravity Core, or external frameworks).Unlike standard logging tools, Sleipnir is an active supervisor: it ingests real-time, non-blocking telemetry streams from running agents by default, while acting as an authoritative centralized policy clearinghouse that evaluates, approves, denies, or mutates high-risk agent tool execution payloads before they are committed to the host system.2. System Architecture & Component OverviewSleipnir splits its responsibilities cleanly between a central UI/Policy broker runtime and an ultra-lightweight instrumentation SDK integrated directly into target agents.2.1 Component BreakdownThe Sleipnir Core Engine (Rust/Tokio): Runs the main event-loop, binds to local Inter-Process Communication (IPC) channels (Unix Domain Sockets or Windows Named Pipes), loads the global policy configurations, and manages asynchronous tasks.The Policy Evaluation Engine: A standalone module within the Core Engine that parses a hot-reloadable Policy.toml. It evaluates arriving JSON tool payloads using regex matchers, cost metrics, and structural validation rules, deciding instantly whether to PassThrough or Escalate.The TUI Frontend (ratatui + crossterm): A zero-allocation terminal user interface structured into an active, 4-quadrant layout that displays streaming context and catches keyboard interrupts.The Sleipnir Agent Interceptor Client (SDK/Crate): A tiny, reusable client library dropped into your Python or Rust agents. It replaces or wraps standard tool execution modules, shipping proposed actions to Sleipnir and awaiting approval flags over the IPC boundary when required.3. execution), and consuming (token telemetry) instantly on one layout without grepping massive log dumps.UC-2: Dynamic Hotkey Pause & HijackAs a operator, I want to press a dedicated hotkey (F5) to (Gatekeeping)As a security engineer, I want the system to seamlessly let harmless tools run while automatically locking up and alerting me if an agent attempts a high-risk operation (e.g., executing a system configuration script or modifying sensitive files).UC-4: Inline Tool MutationAs an operator, when an agent proposes an almost-correct command that trips a policy gate, I want to edit the command arguments natively inside my console and return the corrected payload down the pipeline.UC-5: Instantly Trigger Chandrian AuditsAs an adversarial engineer, I want a single-button macro (Ctrl+C) inside Sleipnir that halts the active agent run and automatically clones its entire state payload straight into Chandrian for immediate vulnerability testing.3.2 Detailed Functional Feature Non-Functional Requirements & Constraints4.1 Performance & Resource OverheadZero-Cost Rendering UI: Sleipnir's terminal engine must consume less than 1% CPU when idling and less than 3% during peak streaming loops. It must avoid wasteful screen-refresh redraws unless state frames are received.Ultra-Low Latency Overhead: For passive, unblocked actions, the IPC transport and structural parsing overhead must be sub-millisecond ($<1\text{ms}$).Memory Footprint: The absolute memory ceiling of the compiler binary runtime must remain below 30MB, leaving maximum memory overhead open for running local models.4.2 Constraints & Operational BoundariesLocal-First Architecture: Sleipnir is explicitly decoupled from standard network web requirements. It operates strictly within a localized sandboxed context via local sockets. No telemetry data or sensitive agent code blocks may pass to external internet interfaces unless explicitly authorized by sub-modules.Single Machine Topology: Initially targeted to manage multiple local concurrent agent tasks executing on the immediate hardware loop. Networked cluster discovery is explicitly out of scope for Phase 1.Platform Support: Native compliance across POSIX systems (Linux/macOS) with target compatibility paths for Windows environments via standard abstractions.5. System State Flow: The Lifecycle of an ActionThe operational logic of Sleipnir handles every execution request through a tight decision loop to optimize safety and processing speed:Generation: Agent creates an action intent (e.g., calling execute_bash with rm -rf ./cache).Intercept & Transport: The integrated agent client intercepts the action, serializes it into a standard JSON metadata block, and drops it into the local socket.Broker Assessment: Sleipnir consumes the frame and checks it against the active parsed rule definitions:Path A (Pass): Action meets clean rules. Sleipnir updates the TUI metrics and sends an unblocked execution acknowledgment immediately.Path B (Escalate): Action trips a verification hook. Sleipnir flags the active running state, sounds a terminal alert, locks the control cursor focus into the dashboard input panel, and waits for explicit key commands (y / n / e).Feedback Execution Loop: The user choice is packaged back over the pipeline channel to direct the agent outcome.
